# Task Service

Сервис для управления задачами с HTTP API на Go.

## Требования

- Go `1.23+`
- Docker и Docker Compose

## Быстрый запуск через Docker Compose

```bash
docker compose up --build
```

После запуска сервис будет доступен по адресу `http://localhost:8080`.

Если `postgres` уже запускался ранее со старой схемой, пересоздай volume:

```bash
docker compose down -v
docker compose up --build
```

Причина в том, что SQL-файл из `migrations/0001_create_tasks.up.sql` монтируется в `docker-entrypoint-initdb.d` и применяется только при инициализации пустого data volume.

## API

Базовый префикс API:

```text
/api/v1
```

Маршруты:

- `POST /api/v1/tasks` — создать одну задачу
- `GET /api/v1/tasks` — список задач
- `GET /api/v1/tasks/{id}` — получить задачу по ID
- `PUT /api/v1/tasks/{id}` — обновить задачу
- `DELETE /api/v1/tasks/{id}` — удалить задачу
- `POST /api/v1/tasks/generate` — сгенерировать задачи по правилу периодичности

## Фича: периодичность задач

### Как это работает

`POST /api/v1/tasks/generate` принимает шаблон задачи (title, description, status) + правило периодичности + окно дат (`start_date` / `end_date`). В ответ возвращается массив созданных задач — по одной на каждую дату, попадающую под правило.

Каждая созданная задача — полноценная запись в таблице `tasks` с полем `scheduled_at`, указывающим на конкретную дату.

### Типы периодичности

| type | Описание | Дополнительные поля |
|---|---|---|
| `daily` | Каждые N дней | `interval` (>= 1) |
| `monthly` | Определённое число каждого месяца | `day_of_month` (1–30) |
| `specific_dates` | Только на указанные даты | `dates` (список YYYY-MM-DD) |
| `even_odd` | Чётные или нечётные числа месяца | `even_days` (true = чётные) |

### Примеры запросов

**Каждые 2 дня:**
```json
{
  "title": "Ежедневный обзвон пациентов",
  "start_date": "2026-04-01",
  "end_date": "2026-04-30",
  "recurrence": { "type": "daily", "interval": 2 }
}
```

**15-го числа каждого месяца:**
```json
{
  "title": "Ежемесячная инвентаризация",
  "start_date": "2026-01-01",
  "end_date": "2026-12-31",
  "recurrence": { "type": "monthly", "day_of_month": 15 }
}
```

**Только по чётным числам:**
```json
{
  "title": "Формирование отчётности",
  "start_date": "2026-04-01",
  "end_date": "2026-04-30",
  "recurrence": { "type": "even_odd", "even_days": true }
}
```

**На конкретные даты:**
```json
{
  "title": "Еженедельная планёрка",
  "start_date": "2026-04-01",
  "end_date": "2026-04-30",
  "recurrence": {
    "type": "specific_dates",
    "dates": ["2026-04-07", "2026-04-14", "2026-04-21", "2026-04-28"]
  }
}
```

### Принятые решения и допущения

- **Генерация = создание реальных задач.** Каждый вызов `/generate` создаёт конкретные записи в БД, а не хранит абстрактное правило. Это упрощает работу с задачами дальше — их можно редактировать, менять статус и удалять независимо друг от друга.
- **`scheduled_at` — дата без времени.** Задачи в трекере привязаны к дню, а не к конкретному времени. Поле хранится как `DATE` в PostgreSQL.
- **`day_of_month` ограничен 1–30** (не 31), чтобы правило работало корректно для всех месяцев. Если в месяце меньше дней, чем указано (например, февраль и `day_of_month=30`), задача создаётся на последний день месяца.
- **`specific_dates` фильтруются по окну `[start_date, end_date]`.** Даты вне окна молча игнорируются.
- **Пустой результат — не ошибка.** Если ни одна дата не попала в окно, возвращается пустой массив `[]` со статусом 201.
- **Вставка в транзакции.** `CreateBatch` оборачивает все INSERT в одну транзакцию — либо все задачи создаются, либо ни одна.
